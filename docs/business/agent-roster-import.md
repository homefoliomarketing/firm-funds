# Agent Roster Import

_Last updated: 2026-06-09_

How admins bulk-import a brokerage's agents from a spreadsheet. The importer lives behind the admin portal (Brokerages page) and runs as a server action, not a REST route.

## Where it lives

| Piece | Location |
|-------|----------|
| Server action | `bulkImportAgentsRoster` in `lib/actions/admin-actions.ts` |
| Parsing helpers | `lib/roster-import.ts` (unit-tested in `lib/roster-import.test.ts`) |
| Upload UI | `app/(dashboard)/admin/brokerages/page.tsx`: the create-brokerage form's "Agent Roster (optional)" box, and the per-brokerage "Import Roster" button |
| Branded roster sent to brokerages | `Firm-Funds-Agent-Onboarding-Roster.xlsx` (generated by `scripts/build-onboarding-roster.py`) |
| Public xlsx template | `public/firm-funds-agent-import-template.xlsx` |
| CSV template | Generated client-side by the "Template" / "Download Template" buttons |

Access is gated by the `agent.invite` capability, held by the Owner and Manager staff tiers only (General Staff does not have it; see `lib/access.ts`).

## Accepted formats and limits

- `.csv` (max 256KB) and `.xlsx` / `.xls` (max 1MB). The 1MB cap is the ONLY bound on zip decompression work (SheetJS inflates the whole archive up front), so do not raise it casually.
- Excel parsing uses SheetJS (`xlsx` 0.20.3, pinned from the SheetJS CDN; see the gotcha in [conventions.md](../development/conventions.md)). Only the FIRST worksheet is read, so the branded roster's "How to Use" sheet is ignored.
- The agent table must sit within the first 2,000 spreadsheet rows (`XLSX_SHEET_ROW_LIMIT`). Longer sheets are rejected with an explicit error, never silently truncated.
- Maximum 200 agents per import (`MAX_ROSTER_ROWS`).

## Parsing pipeline

Both formats are reduced to the same rows-of-strings shape and flow through one pipeline (`rowsToBulkAgents`), so CSV and Excel behavior cannot drift:

1. **Header detection.** The first row among the first 25 non-blank rows containing both "first" and "email" becomes the header row. This tolerates title banners and the Brokerage Details block above the table in the branded roster (its header sits on spreadsheet row 12).
2. **Formula-injection guard.** Any data cell starting with `=`, `+`, `-`, or `@` is rejected (`hasSpreadsheetFormulaRisk`), except phone-shaped values in phone-like columns (leading `+` is normal there).
3. **Column mapping.** Headers are matched fuzzily by normalized substring (for example `RECO Registration #` maps to `recoNumber`, `Email Address *` maps to `email`). Unknown columns like `#`, `Status`, and `Notes` are simply ignored. The street-address matcher explicitly excludes mail-ish headers so "Email Address" cannot be mistaken for an address column.
4. **Blank-row cleanup.** Rows with no value in any mapped column are dropped. The branded roster pre-fills `#` and `Status` on every styled row, so a half-filled roster would otherwise produce dozens of empty "agents". Each surviving row carries its original 1-based spreadsheet row (`sourceRow`), so per-row error messages point at the row the user actually sees in Excel.
5. **Import.** `bulkImportAgents` validates each row (first and last name required; email validated when present), skips duplicates against the brokerage's existing non-archived agent emails, and reports `imported` / `skipped` counts plus per-row errors back to the UI. Existing agents without emails are fine (`agents.email` is nullable by design). Active agent emails are also globally unique (`agents_email_active_key`), so a row whose email already exists at ANOTHER brokerage fails with a per-row error rather than blocking the rest of the import.

Imports are audit-logged as `agent.bulk_import`.

## Notes

- Email is TEMPORARILY optional on import for testing (many real rosters arrive phone-only). Revert before go-live; the markers are in `lib/roster-import.ts` and `lib/validations.ts`.
- The upload travels through the server action body. `next.config.ts` sets `serverActions.bodySizeLimit: '25mb'`, far above the file caps here.
